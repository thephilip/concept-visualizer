## What this one-pager covers

The DNS resolution path for a pod running in an ARO cluster making an external DNS query:

Pod → CoreDNS → node `/etc/resolv.conf` → local dnsmasq → Azure default DNS (168.63.129.16) → internet

And the optional branch for internal domain resolution:

CoreDNS → custom DNS servers (via `dns.operator/default` `spec.servers[]`)

Key failure mode documented: dnsmasq's per-domain 40-second blocking behavior following a single lost upstream query, and the CoreDNS 6-second timeout that surfaces it as repeated SERVFAILs.

## What was explicitly verified vs. assumed

**Verified against official sources:**
- dnsmasq is ARO-specific (KCS 7015390 states this explicitly)
- The 40-second block and 6-second CoreDNS timeout (KCS 7015390)
- 168.63.129.16 as the Azure DNS IP (MS Learn)
- ROSA has no dnsmasq hop (ROSA DNS Operator docs, cluster-dns-operator GitHub)

**Assumed / not independently verified:**
- CoreDNS ClusterIP example of `172.30.0.10` — this is the OCP convention (10th address in service CIDR) but the actual value depends on the cluster's service CIDR

## Known gaps / future updates

- Egress lockdown clusters: the one-pager notes that gateway domains should not be added to custom DNS forwarders, but does not enumerate the specific gateway domains (they are cluster-specific and retrieved via `oc get cluster cluster -o jsonpath="{.spec.gatewayDomains}"`).
- Azure DNS Private Resolver: noted as optional but not documented in detail. A separate one-pager on private ARO DNS architecture may be warranted.
