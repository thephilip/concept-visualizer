# ARO conventions

Platform: **Azure Red Hat OpenShift (ARO)**  
Managed by: Red Hat + Microsoft (joint responsibility model)

Read this file for any one-pager covering ARO-specific architecture, behavior, or failure modes. Read it alongside `../../guide.md`, not instead of it.

---

## What makes ARO different from OCP and ROSA

ARO runs on Azure and is jointly managed by Red Hat and Microsoft. The key architectural differences that affect one-pager accuracy:

| Concern | ARO | ROSA | Self-managed OCP |
|---|---|---|---|
| Node-level DNS forwarder | dnsmasq (ARO-provisioned) | None — CoreDNS → VPC resolver directly | None by default |
| Cloud DNS upstream | 168.63.129.16 (Azure magic IP) | 169.254.169.253 (AWS VPC resolver) | Varies |
| Control plane management | Red Hat + Microsoft SRE | Red Hat SRE (HCP) / customer (Classic infra) | Customer |
| Custom DNS config path | VNet DNS setting + MachineConfig | DHCP option set + Route 53 Inbound Resolver | Installer config |
| Egress lockdown | ARO gateway proxy (optional) | Not equivalent feature | Not applicable |
| Cluster identity | Service Principal or Managed Identity | AWS IAM roles (STS/OIDC) | Varies |

## DNS architecture (ARO-specific)

This is the most common source of misscoped content. Confirm these facts before authoring any ARO DNS one-pager.

- **dnsmasq runs on every ARO node** as an ARO-provisioned service. It is not present on ROSA or self-managed OCP on AWS.
- CoreDNS forwards external queries to dnsmasq via the node's `/etc/resolv.conf` (SystemResolvConf mechanism).
- dnsmasq forwards to the Azure VNet DNS resolver at `168.63.129.16` by default, or to customer-configured resolvers set on the Azure VNet.
- **dnsmasq timeout behavior:** a single lost upstream query causes dnsmasq to block retries for that domain + record type for 40 seconds. CoreDNS has a 6-second timeout, so it returns SERVFAIL to callers for the full 40-second window. See KCS 7015390.
- `/etc/resolv.conf.dnsmasq` — generated by NetworkManager; this is what dnsmasq reads for upstream IPs, distinct from `/etc/resolv.conf` which CoreDNS reads.
- The workaround for the 40-second issue is to forward specific zones directly to `168.63.129.16` via `dns.operator/default` `spec.servers[]`, bypassing dnsmasq.
- Custom DNS: configured at the Azure VNet level + ARO MachineConfig. See MS Learn: `howto-custom-dns`.
- `reddog.microsoft.com` search domain: injected by Azure DHCP. Can cause excessive DNS lookups on custom DNS servers. See KCS 7075586.

**Key diagnostic files on ARO nodes:**
- `/etc/resolv.conf` — points to node IP / 127.0.0.1 (dnsmasq)
- `/etc/resolv.conf.dnsmasq` — upstream resolvers dnsmasq uses
- `journalctl -u dnsmasq` — dnsmasq query log

## Networking

- **SDN → OVN-Kubernetes migration:** ARO clusters migrate network plugins as part of OCP upgrades. Migration is managed but customers must not remove the migration annotation prematurely.
- **Egress lockdown:** Private ARO clusters can have an ARO gateway proxy that intercepts and overrides certain DNS records and egress routes. Do not add gateway domains to custom DNS forwarder entries in the DNS operator.
- **NSG / Azure Firewall:** External DNS failures can be caused by NSG rules blocking outbound UDP/TCP 53, not by in-cluster DNS misconfiguration. Always check the network perimeter in addition to CoreDNS logs.
- `168.63.129.16` is only reachable from within the Azure VNet. It is not a public IP.

## Identity and credentials

- ARO uses a **Service Principal** (legacy) or **Managed Identity** for Azure resource access.
- Credential rotation for Service Principals requires ARO RP backend coordination. See KCS for `InvalidPrincipalId` / `Application.ReadWrite.OwnedBy` issues.
- Pull secret updates: customer-managed in the cluster; ARO does not auto-rotate.

---

## Managed resource group restrictions

- ARO creates and manages a resource group for cluster infrastructure. Customers cannot modify resources in this group directly — Azure deny assignments block it.
- SRE intervention is required for operations inside the managed resource group (e.g. orphaned Azure File shares, disk operations).

---

## Upgrade and lifecycle

- ARO follows OCP lifecycle. Check ARO-specific upgrade paths — not all OCP z-stream edges are available in ARO on the same schedule.
- For upgrade pre-checks, use the ARO-specific proactive case process and provide the Azure Resource ID: `az aro show -n <name> -g <rg> --query id`
- Custom domain cert rotation: ARO does not auto-rotate certs for custom domains. See KCS 6458661.
- EOL OCP versions: MachineConfigPool and network operator issues are common on clusters running end-of-life versions.

## Identifying an ARO cluster in a must-gather

```bash
# Check if cluster is ARO
oc get cm cluster-config-v1 -n kube-system -o yaml | grep -A2 "azure:"
# Expect: aro: true

oc get cm openshift-install-manifests -n openshift-config -o yaml | grep invoker
# Expect: invoker: ARO
```

## Scope banner text (copy-paste)

Use this in any one-pager that covers ARO-specific content:

> **ARO-specific.** [The behavior/component described here] applies only to Azure Red Hat OpenShift. On ROSA and self-managed OCP, [brief description of the difference].

Example (DNS):
> **ARO-specific.** The dnsmasq node-level forwarder and the 40-second timeout/blocking behavior described here apply only to Azure Red Hat OpenShift. On ROSA and self-managed OCP, CoreDNS forwards external queries directly to the VPC/cloud DNS resolver — no dnsmasq hop exists and this failure mode does not apply.
